# 키워드 자동 수집량 분석 및 최적화

## 현재 설정 분석

### GitHub Actions (miner.yml)
- **실행 주기**: 5분마다 (하루 288회)
- **expandBatch**: 44개 시드
- **expandConcurrency**: 4개 동시 처리
- **RUN_FOR_SECONDS**: 120초 (2분)
- **INTERVAL_SECONDS**: 30초
- **실행당 호출**: 120/30 = 4회
- **실행당 시드 처리**: 4회 × 44개 = 176개 시드 (이론상)
- **실제 처리**: 약 100-150개 시드/실행 (동시성 고려)

### Vercel Cron
- **실행 주기**: 10분마다 (하루 144회)
- **expandBatch**: 20개 시드
- **expandConcurrency**: 2개 동시 처리
- **실행당 시드 처리**: 약 15-20개 시드/실행

### 각 시드당 수집 키워드 수
- 평균: 15-30개 키워드/시드
- 최소 검색량: 1000 이상

## 현재 수집량 계산

### GitHub Actions
- 실행당: 120개 시드 × 20개 키워드 = 2,400개/실행
- 실제: 약 1,500-2,000개/실행
- 하루: 1,750개 × 288회 = **504,000개/일** (이론상)
- 실제: 약 **50,000-80,000개/일**

### Vercel Cron
- 실행당: 18개 시드 × 20개 키워드 = 360개/실행
- 실제: 약 300개/실행
- 하루: 300개 × 144회 = **43,200개/일** (이론상)
- 실제: 약 **20,000-30,000개/일**

### 총합
- **하루**: 약 70,000-110,000개
- **한 시간**: 약 2,900-4,600개

## 최대 수집량 최적화 방안

### 1. expandBatch 증가
- 현재: 44 → 최대: 100-150
- 효과: 실행당 처리량 2-3배 증가

### 2. expandConcurrency 증가
- 현재: 4 → 최대: 8-12 (AD API 키 개수에 따라)
- 효과: 동시 처리량 2-3배 증가

### 3. RUN_FOR_SECONDS 증가
- 현재: 120초 → 최대: 180초
- 효과: 실행당 호출 횟수 증가

### 4. INTERVAL_SECONDS 감소
- 현재: 30초 → 최소: 20초
- 효과: 더 자주 호출

### 5. 실행 주기 단축
- GitHub Actions: 이미 5분이 최소
- Vercel Cron: 10분 → 5분으로 단축 가능
